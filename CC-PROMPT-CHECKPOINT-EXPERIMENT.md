**The problem we're documenting:** The ideal recursive covenant for VTXO refresh — `OP_INSPECTINPUTSCRIPTPUBKEY == OP_INSPECTOUTPUTSCRIPTPUBKEY` (7 bytes: `00d100ca7b8887`) — doesn't work because Ark's `buildOffchainTx` wraps every VTXO input in a 2-leaf checkpoint transaction. The Introspector evaluates the Arkade Script against the arkTx, where `OP_INSPECTINPUTSCRIPTPUBKEY` returns the **checkpoint's** witness program (2-leaf taptree), not the **original VTXO's** witness program (3-leaf taptree). So `input == output` always fails.

The current workaround is a 4-byte refresh script (`00d15188`) that checks output taproot version only. This works but introduces an agent-integrity dependency: the agent process must construct the refresh transaction to the same address. An attacker with sustained agent process access could redirect refresh outputs to a different taproot address.

**We are writing a technical brief for Ark Labs (Tiero) to propose a fix.** The brief must include regtest evidence, not hypotheticals.

### Goal 2: Write the Technical Brief

After Goal 1 completes and you have the experiment output, create `docs/ARK-RECURSIVE-COVENANT-BRIEF.md`. This is a technical document for Ark Labs engineers. **Structure:**

#### 1. Problem Statement (keep to 1 paragraph)
Recursive covenants on Ark — where a script enforces `input.scriptPubKey == output.scriptPubKey` — are broken by checkpoint wrapping. This affects any application using `OP_INSPECTINPUTSCRIPTPUBKEY` in Arkade Script. The limitation is architectural, not a bug in the Introspector or arkd individually.

#### 2. Background: What We're Building (2-3 paragraphs, max)
Golem is a Bitcoin agent wallet on Ark. Agents receive Lightning payments via L402 and hold funds in covenant-secured VTXOs. The agent operates with zero private key material — claims use preimage + Introspector, refresh uses covenant + Introspector. The user's private key stays on their mobile device.

The covenant VTXO is a 3-leaf taptree: Leaf 0 (refresh via Introspector), Leaf 1 (collaborative: alice + server), Leaf 2 (unilateral exit: CSV + alice). All validated on regtest with full lifecycle: claim → refresh → consolidation (2→1) → collaborative spend.

Link to the Golem covenant module: `src/covenant/` (6 modules, ~300 lines total).

#### 3. The Limitation (technical, with evidence)
Include the EXACT output from Experiments A and B. Show:
- What `buildOffchainTx` does to inputs (wraps in 2-leaf checkpoint)
- What `OP_INSPECTINPUTSCRIPTPUBKEY` returns (checkpoint WP)
- What the recursive covenant expects (VTXO WP)
- Why they differ (checkpoint taptree ≠ VTXO taptree)

Include the 7-byte script bytecode (`00d100ca7b8887`) and explain what each byte does.

Include the 4-byte workaround bytecode (`00d15188`) and explain why it's insufficient: it checks taproot version only, so any taproot address passes, creating an agent-integrity dependency for refresh destination enforcement.

#### 4. Impact Beyond Golem
This limitation affects EVERY application that wants to use `OP_INSPECTINPUTSCRIPTPUBKEY` on Ark:
- **Keyless agent wallets** (Golem's use case): can't enforce refresh-to-same-address without agent trust
- **Merchant receive-only addresses**: can't build "funds can only move to my cold storage" covenants
- **Automated treasury management**: can't enforce "consolidate to same script" policies
- **Any recursive covenant**: the fundamental primitive `input == output` is broken by checkpoint wrapping

Frame this as: the Introspector and Arkade Script VM support a powerful set of introspection opcodes, but checkpoint wrapping means `OP_INSPECTINPUTSCRIPTPUBKEY` returns unexpected values in the most common use case (spending a VTXO). This limits the practical utility of input introspection to applications that don't care about the input's original script.

#### 5. Proposed Solutions (three proposals, each with pros/cons)

**Proposal A: New opcode `OP_INSPECTORIGINALSCRIPTPUBKEY`**

A new Arkade Script opcode that resolves through the checkpoint wrapper to return the original VTXO's scriptPubKey. The existing `OP_INSPECTINPUTSCRIPTPUBKEY` keeps its current literal semantics. Script authors opt into trace-through explicitly by using the new opcode.

Pros:
- No behavior change to existing opcodes — fully backward compatible
- Script authors explicitly choose whether they want checkpoint-resolved or literal values
- Clean semantic distinction: "what's literally on the input" vs "what was the original VTXO"

Cons:
- New opcode requires Introspector to understand checkpoint structure (coupling between Introspector and arkd's internal wrapping)
- If checkpoint format changes, the unwrapping logic must be updated
- Added complexity in the Introspector's opcode evaluation

**Proposal B: Introspector Packet extension — carry original VTXO witness program**

Extend the OP_RETURN Introspector Packet (TLV format, magic `0x41524b`) to include a new TLV record type that carries the original VTXO's witness program for each input. The Introspector already parses these packets — adding a new record type is a well-defined extension point. `OP_INSPECTINPUTSCRIPTPUBKEY` (or a new opcode) could read from this record instead of from the transaction input.

The packet format is already versioned via TLV. Current record type `0x01` carries Arkade Script entries. A new record type (e.g., `0x02`) could carry input VTXO metadata.

Pros:
- Uses the existing Introspector Packet extension mechanism (TLV)
- No coupling to checkpoint internal structure — the transaction builder (who knows the original VTXO) includes the data explicitly
- The Introspector doesn't need to "unwrap" anything — it reads what's declared in the packet
- Easily auditable: anyone can verify the declared WP matches what they expect

Cons:
- Trust question: who provides the original WP, and can they lie? If the builder provides it, a malicious builder could declare a wrong WP. The Introspector would need to verify it against the checkpoint's wrapped VTXO (which re-introduces some checkpoint awareness).
- Slightly larger OP_RETURN (32 bytes per input for the WP)
- New packet record type needs spec and implementation

**Proposal C: `buildOffchainTx` preserves VTXO script in checkpoint**

Modify `buildOffchainTx` so that the checkpoint transaction includes the original VTXO's taptree (or a commitment to it) in a way that makes the checkpoint's witness program match the VTXO's. For example, the checkpoint could include the original VTXO script as an additional leaf, or the checkpoint's key-path could be tweaked with the original VTXO's script hash.

Pros:
- Fixes the problem at the source — no Introspector changes needed
- `OP_INSPECTINPUTSCRIPTPUBKEY` returns the correct value naturally
- Simplest mental model: "the checkpoint IS the VTXO from the Introspector's perspective"

Cons:
- Modifies Ark's core transaction construction — highest risk of unintended consequences
- Checkpoint structure serves multiple purposes (server forfeit, unilateral settlement). Adding VTXO data to checkpoints could interact with these in unexpected ways
- Every VTXO script variant would need to be carried through checkpoint construction — increased complexity in `buildOffchainTx`
- May break existing checkpoint assumptions in arkd's settlement logic

#### 6. Our Recommendation

We recommend **Proposal A** (new opcode) as the cleanest solution, with **Proposal B** (packet extension) as a pragmatic alternative if adding a new opcode is too heavy.

Proposal C touches Ark's core transaction construction and carries the highest risk. We'd defer to Ark Labs' judgment on whether it's viable, but we think A or B achieve the same goal with less risk to existing infrastructure.
